# ADR-0003 — Adopt SonarCloud for continuous code quality analysis

## Version

1.0.0

## Date

2026-05-02

## Status

Accepted

## Context

As Seal UI grows into a reusable design system consumed across multiple applications, maintaining a consistent bar of code quality becomes increasingly important. Without automated quality gates, issues such as duplicated logic, insufficient test coverage, and potential bugs can accumulate silently between releases. The project already enforces static analysis via `flutter analyze` and `flutter_lints`, but these tools operate only locally and do not track trends over time, highlight regressions across pull requests, or provide a centralised view of quality metrics. A hosted quality platform that integrates with the CI pipeline and provides visibility into coverage, duplication, and code smells was identified as the missing piece.

## Decision

We will use SonarCloud as the continuous code quality platform for Seal UI. SonarCloud is free for open-source projects and integrates natively with GitHub Actions via the `SonarSource/sonarcloud-github-action`. The project is configured through `sonar-project.properties`, pointing analysis at `lib/` as the source root and `test/` as the test root. Coverage data is fed from LCOV reports generated by `flutter test --coverage`, which are produced in CI before the Sonar scan runs. The quality gate enforces thresholds on new code for coverage, duplicated lines, and reliability rating, ensuring that every pull request is evaluated before merge.

## Consequences

**Positive:**
- Provides a persistent, visual record of quality trends across every commit and pull request
- Automatically detects duplicated blocks and code smells that `flutter analyze` does not surface
- Coverage tracking is tied to CI, preventing coverage regressions from reaching the main branch
- The quality gate acts as an additional merge check alongside tests and linting, making the bar for contributions explicit and consistent
- Free for open-source projects with no infrastructure to maintain

**Negative:**
- Adds a dependency on an external SaaS platform; analysis results are unavailable if SonarCloud experiences downtime
- LCOV report generation requires `flutter test --coverage` to run in CI on every push, slightly increasing pipeline duration
- Dart support in SonarCloud is community-driven and may lag behind Dart language updates or miss language-specific patterns
- New contributors must be aware of the quality gate to understand why a pull request is blocked when metrics regress

The `sonar-project.properties` file at the repository root centralises all configuration. The `sonar.dart.lcov.reportPath` property points to `coverage/lcov.info`, which is generated by the coverage step that precedes the scan in the CI workflow.
